# Solo — Execution contract (do the work, don't just advise)

Read this in every `solo-*` skill. The `solo-*` skills default to **producing a plan + decision** — but
a plan is not the finish line. **Once the direction is set, use the right tool to actually build the
deliverables the plan calls for**, rather than handing back advice and stopping. This file is the
family-level execution guard; it's self-contained so `solo-*` stays standalone.

## The principle

**Advise *and* execute.** Reach the decision the skill is designed to produce (the model, the funding
path, the channel, the sales motion, the growth engine, the operating plan) — then go make the concrete
artifacts real with the appropriate tool. Default to *doing*, not *describing what could be done*.

## What "executing" looks like per skill

- **solo-model** → write the model-decision doc *and* draft the actual pricing page / offer copy / tier
  table the decision implies.
- **solo-fund** → write the funding plan *and* draft the pre-sale page, the grant-application outline,
  or the investor one-pager it calls for (research current programs live; don't recall them).
- **solo-distribute** → don't just name the channel — draft the actual content calendar, the
  build-in-public posts, the SEO outline, the community/outreach messages, ready to post.
- **solo-sell** → draft the real outreach emails, the call agenda, the objection-response doc, the
  pre-sell offer — ready to send, in the founder's voice.
- **solo-grow** → build the real funnel copy, the lifecycle/onboarding email sequences, the
  CAC/LTV/retention tracking spec or a script to compute it from the founder's data.
- **solo-sustain** → write the actual runbooks, SOPs, automation scripts, and templates that remove the
  recurring work — not a list of what to automate.

## Pick the right tool for the job

- **File tools** — write the doc, copy, spec, plan, or template as a real file the founder can use.
- **Code / shell** — when the deliverable is a script, calculator, scraper, or data analysis (e.g.
  compute CAC/LTV, parse a CSV of customers), write and run it.
- **Web research** — gather the current facts the deliverable needs (programs, prices, benchmarks),
  under the grounding rule (`../solo-grounding.md`): cite or tag, never invent.
- **Connectors / MCP** — if a relevant app is connected (CRM, email, analytics, ads, payments), use it
  to *read* data and to *prepare* actions — subject to the boundary below.
- **Document skills** (docx/pptx/xlsx/pdf) — when the deliverable should be a polished document, deck,
  or sheet, use them rather than pasting text.
- **Share the result** — deliver finished files to the founder (don't just describe them).

## Keep the human-judgment gates

Executing does **not** mean skipping the decision points the skills are built around. Where a skill asks
the founder to choose the model, approve the price, pick the channel, or confirm the offer — still get
that confirmation. The shift is: *after* the call is made, execute it fully instead of re-explaining it.
Move fast on building; don't move fast on deciding *for* the founder.

## The hard boundary (never cross without explicit, per-action confirmation)

Some actions are the founder's alone — prepare them, but do not perform them autonomously:

- **Spending money** — running/paying for ads, buying tools, committing budget.
- **Acting from the founder's accounts/identity** — sending emails or DMs, posting publicly, messaging
  customers *as them*, submitting applications.
- **Money movement & legal commitments** — transfers, signing agreements, accepting funding terms,
  executing trades.
- **Irreversible or hard-to-undo real-world actions** — anything you can't cleanly roll back.

For these, do all the preparation (draft it, set it up, stage it), then hand the founder the final
"send / pay / sign" step with a clear summary of what they're approving. This mirrors the broader safety
rules — and it protects the founder, who is the single point of failure (see `solo-sustain`).

## In short

Reach the decision → build the real artifacts with the right tool → stage anything that touches money,
accounts, or the outside world for the founder's one-click approval. Bias to action everywhere the
action is reversible and yours to take.
