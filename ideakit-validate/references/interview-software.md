# Interview Question Bank — Software Product Mode

Use this question bank during Phase 1 when the seed is a software product, feature, tool, or
automation. Walk through themes in order. Adapt — drill down on vague answers, skip questions
already answered, surface contradictions when you hear them.

Aim for 6–12 total questions across the conversation. Quality over coverage.

## Theme 1 — The Problem (start here)

The job here is to make sure the user is solving a real, specific problem — not building a
solution in search of one.

- "What is the actual problem you're trying to solve? Walk me through a recent moment when you
  felt this pain — or saw someone else feel it."
- "How is this problem being solved today? (Manually? With another tool? Not at all?)"
- "What's the cost of the problem staying unsolved? (Time? Money? Frustration? Lost revenue?)"
- "Why now? What changed that makes this worth building today vs. a year ago?"

**Drill-down triggers:**
- If the user describes a solution, not a problem → "What's the underlying problem that solution addresses?"
- If the answer is theoretical → "Can you give me a concrete example from the last week?"

## Theme 2 — The User

- "Who specifically would use this? Be as narrow as you can — name a person if possible."
- "How many of these people exist, roughly? Is this a niche or a broad market?"
- "Are you the user, or are you building this for someone else? If for someone else, how have
  you talked to them about it?"
- "What does this user use today instead?"

**Drill-down triggers:**
- "Everyone" → "If you had to pick the first 10 users, who would they be specifically?"
- User is a hypothetical → "Have you talked to one? What did they actually say?"

## Theme 3 — The Value

- "If this thing existed and worked perfectly, what would change for the user? What can they
  do that they couldn't before — or do faster, cheaper, or better?"
- "What's the one thing this absolutely must do well to be worth using?"
- "What would make a user recommend this to a colleague?"

## Theme 4 — Constraints & Context

- "What's your timeline? Is this a weekend hack, a 1-quarter project, or something open-ended?"
- "What's the budget — your time, money, or both?"
- "Are you building this solo, or with a team? What skills are on the team?"
- "Are there constraints that aren't obvious — compliance, existing systems to integrate with,
  data you do or don't have access to?"

## Theme 5 — Definition of Done (v1)

- "What's the smallest version of this that would prove the idea works?"
- "What's explicitly NOT in v1?"
- "How will you know it's working? What signal tells you to keep building, vs. stop?"

## Theme 6 — Hidden Assumptions (use sparingly, when needed)

- "What has to be true about the user, the market, or the technology for this to work?"
- "What's the part of this idea you're least sure about?"
- "If a friend told you the same idea, what's the first objection you'd raise?"

## Closing the Interview

When you have enough material, write the **idea snapshot** (see SKILL.md Phase 1 for format) and
ask the user to confirm or correct before moving to research.

If the user said "I don't know" to anything important, list those as **open questions to research**
and explicitly call them out — don't quietly drop them.
